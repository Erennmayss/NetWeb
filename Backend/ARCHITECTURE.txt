CYCLE DE VIE D'UNE ALERTE
═════════════════════════════════════════════════════════════════════════════════════════════════════════

T+0s    Snort détecte une anomalie
        └─ Écrit dans /var/log/snort/alert (serveur Ubuntu)

T+1s    Recuperation.py lit l'alerte
        └─ Parse les infos (IPs, ports, type, sévérité)

T+2s    Envoi parallèle:
        
        A) API Flask:
           • Envoie à /api/alerts (POST)
           • API enregistre dans Supabase
        
        B) Snort Processor:
           • Appel direct insert_alert()
           • Insère dans Supabase
           • Envoie notifications immédiatement:
             - Toast Windows (< 1s)
             - Email SMTP (2-3s)

T+3s    Notifier détecte la nouvelle alerte (s'il n'y a pas d'alerte à T+2s)
        └─ Envoie à nouveau notifications (évitement doublon interne)

T+5s    Prochaine itération du notifier
        └─ Polling Supabase


SÉCURITÉ & DONNÉES
═════════════════════════════════════════════════════════════════════════════════════════════════════════

✅ Supabase: SSL mode REQUIRE (chiffrement en transit)
✅ Email: SMTP Gmail avec clé d'application (pas mot de passe)
✅ Logs: Fichiers locaux (%APPDATA%\IDS_Notifier\)
✅ État: JSON persistant (alertes vues en mémoire + fichier)
✅ Identifiants: Variables d'environnement .env
